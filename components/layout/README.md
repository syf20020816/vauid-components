# React 高性能虚拟布局音视频组件库架构与设计文档

## 1. 行业痛点与现状分析

### 1.1 痛点：布局切换导致的重渲染与 iframe 重载

在传统的 React 音视频会议项目（如基于 LiveKit Components, Agora UIKit, Twilio Video 构建的应用）中，**布局切换导致重渲染是一个非常普遍且棘手的痛点**。
当用户从 Grid（网格）切换到 Focus（演讲者）或全屏时，常规做法是条件渲染不同的组件树（例如 `<GridView />` 替换为 `<SpeakerView />`）。这会导致：

1. **DOM 卸载与重建**：React Diff 算法判定节点类型或 Key 变化，直接销毁旧 DOM 并创建新 DOM。
2. **媒体流中断/重载**：对于 `<video>` 标签，可能导致短暂黑屏或重新缓冲；**对于 `<iframe>`（如共享的网页、3D 模型、外部协作白板），DOM 的卸载会直接导致 iframe 完全重新加载**，丢失所有内部状态，用户体验极差。
3. **性能抖动**：大量 DOM 节点的增删会引发浏览器的重排（Reflow）和重绘（Repaint），在低端设备或大型会议（如 25+ 人）中会导致明显的卡顿。

### 1.2 主流平台解决方案对比

| 解决方案                               | 代表平台/库                   | 原理                                                                               | 优点                                                                           | 缺点                                                                      |
| :------------------------------------- | :---------------------------- | :--------------------------------------------------------------------------------- | :----------------------------------------------------------------------------- | :------------------------------------------------------------------------ |
| **条件渲染 (多套布局)**                | 多数基础 React 组件库         | 切换状态时渲染不同的组件树。                                                       | 实现简单，代码逻辑清晰。                                                       | **iframe/视频必重载**，DOM 开销大，切换有闪烁。                           |
| **绝对定位 + 显隐控制**                | 部分定制化合规项目            | 所有 Tile 都在 DOM 中，通过 `display: none` 或绝对定位控制。                       | 避免了 DOM 销毁，iframe 不会重载。                                             | DOM 节点过多时内存占用高；隐藏的视频仍在解码，消耗 CPU/GPU。              |
| **FLIP 动画技术**                      | `react-flip-toolkit` 等       | 记录布局变化前后的位置 (First, Last)，用 Transform 模拟移动 (Invert, Play)。       | 动画平滑，视觉体验好。                                                         | 复杂布局（如分页、主副屏切换）下计算成本极高，难以处理 DOM 逻辑归属。     |
| **Canvas / WebGL 渲染**                | Zoom Web 客户端, 某些高端方案 | 将视频流绘制到单一 Canvas 上，布局切换仅改变绘制坐标。                             | 极致性能，零 DOM 重渲染，无 iframe 问题（若用 WebGL 渲染网页）。               | 丧失原生 HTML 交互性（右键菜单、无障碍、iframe 嵌入困难），开发成本极高。 |
| **虚拟布局 + Transform 驱动 (本方案)** | **vauid**                     | DOM 节点复用，通过计算引擎预计算位置，切换时仅修改 `transform`，结合智能填补算法。 | **完美保留 iframe 状态**，GPU 加速零重排，DOM 数量可控，性能与交互性完美平衡。 | 引擎开发复杂度较高（需处理坐标计算、多线程、缓存）。                      |

## 2. 核心设计理念

1. **DOM 恒定与虚拟映射**：所有参与会议的 Tile（如 6 个）在 DOM 中始终存在，且物理上呈单列平铺。视觉上的“网格”或“主视口”仅是通过 CSS `transform` 映射出的虚拟位置。
2. **GPU 加速与主动分层**：布局切换不触发 Reflow，仅触发 Composite。通过 `will-change: transform` 主动将 Tile 提升为独立的合成层（Composite Layer），由 GPU 处理位移，实现 60fps 的丝滑切换。
3. **智能位移优化（末尾填补算法）**：当某个 Tile 被提取到右侧主视口时，其在左侧列表留下的空缺，**由当前列表的最后一个 Tile 直接填补**，而非后续所有 Tile 依次前移。这将 Transform 的并发计算量从 $O(N)$ 降至 $O(1)$。
4. **按需渲染与 0 帧策略**：非当前页的 Tile 设置为 `opacity: 0` 且 `pointer-events: none`，并通过暂停 `<video>` 元素（或停止 `requestVideoFrameCallback`）实现“0 帧”渲染，仅保留 AudioContext/WebRTC 连接，极大降低解码压力。
5. **异步计算与缓存**：布局计算剥离至 Web Worker，避免阻塞主线程；计算结果基于 `容器尺寸 + 参与人数 + 布局模式` 进行 Hash 缓存；监听 `ResizeObserver` 实现防抖自动重算。

## 3. 系统架构设计

系统采用分层架构，确保高内聚、低耦合：

```text
┌─────────────────────────────────────────────────────────────────┐
│                       表现层 (UI Components)                     │
│  - StageView (主视口)  - ThumbnailList (左侧列表)                 │
│  - Tile (复用组件，内置 will-change, video/iframe 渲染逻辑)         │
└───────────────────────────────┬─────────────────────────────────┘
                                │ (订阅状态, 接收 Transform 样式)
┌───────────────────────────────▼─────────────────────────────────┐
│                     状态管理层 (State Manager)                  │
│  - 维护当前布局模式 (Grid/Focus/Fullscreen)                     │
│  - 维护分页状态、焦点参与者 ID、可见性状态 (Visible/0-frame)    │
│  - 媒体流控制 (Mute/Unmute, 0-frame 暂停/恢复指令)              │
└───────────────────────────────┬─────────────────────────────────┘
                                │ (触发计算请求)
┌───────────────────────────────▼─────────────────────────────────┐
│                   布局计算引擎层 (Layout Engine)                │
│  - 主线程代理 (Engine Proxy): 接收请求，管理 Cache，分发 Worker │
│  - Web Worker (计算核心): 执行纯函数布局计算，返回坐标矩阵      │
│  - 智能填补算法 (Smart Fill Algorithm): 计算最小位移路径        │
└───────────────────────────────┬─────────────────────────────────┘
                                │ (PostMessage 传递计算结果)
┌───────────────────────────────▼─────────────────────────────────┐
│                     底层能力层 (Core Utilities)                 │
│  - ResizeObserver (容器尺寸监听，带防抖 Debounce)               │
│  - LRU Cache (布局计算结果缓存)                                 │
│  - Weak Network Strategy (弱网降级、分页加载、动态流控策略)     │
└─────────────────────────────────────────────────────────────────┘
```

## 4. 核心引擎设计 (Layout Engine)

### 4.1 引擎输入与输出

- **Input**:
  - `containerSize`: `{ width, height }`
  - `participants`: `[{ id: 'A', isActive: true }, { id: 'B' }, ...]` (长度为 N)
  - `layoutMode`: `'grid' | 'focus' | 'fullscreen'`
  - `pageSize`: 每页显示的 Tile 数量 (如 4)
- **Output**:
  - `layoutMap`: `Map<participantId, { x, y, width, height, zIndex, opacity, isPaused }>`

### 4.2 智能填补算法 (Smart Fill Algorithm) - 核心创新

假设左侧列表容量为 3，当前参与者为 `[A, B, C, D, E, F]`。

- **常规做法**：聚焦 `C` 到主视口，左侧变为 `[A, B, D, E, F]`。此时 `D, E, F` 都需要执行 Transform 向前移动一格。（3 个元素移动）
- **本引擎做法**：聚焦 `C` 到主视口。引擎将 `F`（最后一个元素）的虚拟坐标直接赋给 `C` 原来的位置。左侧视觉呈现为 `[A, B, F, D, E]`（逻辑上 D, E 位置不变，F 填补空缺）。
- **结果**：只有 `C` (移至主视口) 和 `F` (填补空缺) 发生 Transform 变化。**移动开销恒定为 2 个元素**，无论列表多长。

### 4.3 多线程与缓存机制

1. **缓存命中**：主线程在请求计算前，生成 Cache Key：`md5(`${width}_${height}_${layoutMode}_${N}`)`。若命中 LRU Cache，直接返回结果，耗时 `< 1ms`。
2. **Worker 计算**：若未命中，主线程通过 `postMessage` 将参数发送给 Worker。Worker 执行纯数学计算（无 DOM 操作），计算完成后返回坐标矩阵。
3. **Resize 处理**：`ResizeObserver` 监听容器变化，触发 100ms 防抖的重新计算请求，确保窗口缩放时布局自适应。

### 4.4 0 帧与可见性控制策略

对于非当前页的 Tile，引擎输出状态包含 `isVisible: false`。
UI 层接收到此状态后执行：

```css
.tile-hidden {
  opacity: 0;
  pointer-events: none;
  /* 不使用 display: none 或 visibility: hidden，以保持其在 DOM 树中的位置，避免后续移回时重新触发布局计算 */
}
```

同时，组件内部执行：

```javascript
if (!isVisible) {
  videoElement.pause(); // 停止视频解码，实现真正的 "0 帧"，节省 CPU/GPU
  // 保持 WebRTC track 的 audio 部分 enabled，确保声音不断
} else {
  videoElement.play();
}
```

## 5. 需要实现的 Features 列表

### 核心布局与渲染

- [x] **虚拟布局引擎**：支持 Grid, Focus (双栏), Fullscreen 模式的无缝切换。
- [x] **DOM 恒定与 Transform 驱动**：切换布局无 DOM 增删，iframe/视频无重载。
- [x] **智能末尾填补算法**：最小化布局切换时的 Transform 元素数量。
- [x] **GPU 主动分层**：自动注入 `will-change: transform`，优化合成层。

### 性能与资源管理

- [x] **Web Worker 计算**：布局计算 100% 异步，不阻塞主线程 UI 响应。
- [x] **LRU 计算结果缓存**：相同尺寸和人数配置下的计算结果毫秒级返回。
- [x] **自适应 Resize**：内置 `ResizeObserver`，窗口变化时平滑过渡到新布局。
- [x] **智能 0 帧渲染**：非可视区域 Tile 自动暂停视频解码，仅保留音频流，大幅降低多路视频时的设备发热和卡顿。

### 高级特性 (弱网与扩展)

- [x] **虚拟分页 (Virtual Pagination)**：支持海量参与者（如 50+），通过分页机制控制 DOM 节点上限（如始终只挂载 9 个 DOM 节点，通过复用和 Transform 模拟更多）。
- [x] **弱网自适应策略**：与流控模块联动，当检测到弱网时，引擎自动降级布局（如从 Grid 强制切换为 Focus），并触发非主视口 Tile 的分辨率下调或 0 帧处理。
- [x] **灵活的 Slot 渲染**：Tile 组件支持 `children` 注入，完美兼容 `<video>`, `<iframe>`, `<canvas>` 或自定义 React 组件。

| :---------------------- | :------------------------------------- | :--------------------------------------------------------------------------- |
| **iframe/复杂内容支持** | 切换布局必重载，状态丢失，体验断裂。   | **完美保持**，DOM 节点不销毁，iframe 内部状态（如滚动位置、表单）100% 保留。 |

## 7. 目标受众与适用场景 (Who Needs This)

1. **需要嵌入复杂 Web 内容的协作平台**：
   - _场景_：在线协同设计、共享网页浏览、3D 模型评审、内嵌复杂业务系统的远程支持。
   - _需求_：这些场景重度依赖 `<iframe>` 或 `<canvas>`，传统组件库的重载问题是致命伤，本方案是唯一优雅的解法。
2. **对性能和用户体验要求极高的 C 端/B 端会议产品**：
   - _场景_：大型 Webinar、在线教育大班课、高端远程医疗会诊。
   - _需求_：需要平滑的布局切换动画，且在低端设备（如旧款笔记本、平板）上多路视频不能卡顿（依赖 0 帧和 Worker 优化）。
3. **希望快速构建但受限于 LiveKit/Agora 默认 UI 的开发者**：
   - _场景_：使用了 LiveKit/Agora 的底层 SDK，但对其提供的默认 UI 组件（布局僵硬、性能一般）不满意，希望替换为高性能、可深度定制的布局引擎。
4. **需要精细化弱网控制的场景**：
   - _场景_：移动网络环境下的音视频应用。
   - _需求_：需要布局引擎与弱网策略深度绑定，在卡顿时自动降级布局并释放解码资源。
